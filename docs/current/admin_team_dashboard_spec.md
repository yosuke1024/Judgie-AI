# 管理者向けチームダッシュボード表示機能 仕様書

## 概要
管理者がハッカソンに参加している各チームのダッシュボード（評価結果、レーダーチャート、スコア推移、AI審査員からの詳細なフィードバック、対話スレッドなど）を、左ペイン（サイドバー）から直接選択して閲覧できるようにします。

## 要件
1. **左ペイン（サイドバー）でのチーム選択**:
   - 管理者（`admin` または `observer`）ログイン時、サイドバーに「チームダッシュボード」セクションを表示する。
   - 参加しているチーム一覧を動的にロードし、スクロール可能な形で表示する。
2. **ダッシュボードの表示**:
   - URL `/admin/teams/:teamId` で指定のチームのダッシュボードを表示する。
3. **読み取り専用モード**:
   - プロフィールの更新、ファイルのアップロード、Q&A対話スレッドへの新規送信は無効（または非表示）とする。

## Implementation Report

### 実装時の変更点
- **`App.tsx`**: 管理者用の専用ルートとして `/admin/teams/:teamId` を追加。
- **`Layout.tsx`**: 管理者時にチーム一覧を動的に取得し、サイドバーにスクロールエリア（チーム名リンク一覧）をレンダリング。
- **`TeamDashboard.tsx`**:
  - `useParams` を用いてURLから `teamId` を検出した場合に `isReadOnly = true` とする。
  - プロフィール、成果物提出、Q&A送信などを閲覧専用（無効化または非表示）に切り替える。
  - ログインチーム自身の情報ではなく、`teamId` に基づいた評価履歴、対話スレッド等のデータをロードする。
  - 残り相談回数は `evaluations` の履歴から算出し、プロフィールカードの下部に表示。

### 非自明な回避策
- **`max_consultations` と `max_qa_turns`**: 管理者が閲覧している時はログインユーザー（管理者）の属性にこれらの設定が含まれないため、`settingsApi.getProject()` からハッカソン設定をロードして適用しました。
- **`maxQaTurns` のスコープ重複**: `renderEvaluationDetail` 内で `maxQaTurns` が再宣言されていたため、コンパイラエラー（TS6133）が発生。外側のステート定義で統一することで解決しました。

### なぜこの設計を採用したか（Why）
- **コマンドセンターからの分離**: 当初はCommand Centerの一機能としてのダッシュボード閲覧を計画していましたが、ユーザーのフィードバックに基づき、直接サイドバーからチームを選択できるようにしました。これにより、各チームの画面へ1クリックで直感的に切り替えられるようになり、管理・観察のUXが大幅に向上しました。
- **TeamDashboardの再利用（DRY）**: 管理者専用 of ダッシュボード表示画面を新規作成するのではなく、既存 of `TeamDashboard` コンポーネントをProps/URLパラメータ制御で読み取り専用化して再利用することで、コード重複を避け、将来のUI拡張時にも保守しやすくしました。
